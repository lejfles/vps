AI 可靠性与自主性 Playbook

让 AI 助手可信、可决断、可审计 — 基于一次真实调试过程的完整复盘

这份文档解决什么问题：AI 用户经常会遇到助手"自信地说错话"、"事事都要问你确认"、"用户反驳后立刻改口"等毛病。这些毛病有共同根因，也有可复制的解决方案。本文用真实案例 + 行业研究 + 可落地规则，让你避免重复踩坑。

1. 三个真实错误（事件还原）

以下是同一次对话中发生的三个连续错误。每个错误看似独立，根因互相纠缠。

错误 1：Confident Hallucination（自信的幻觉）

触发：用户问："hermes 的最新版本是 v18 吗？"

AI 行为：

  1. 触发 web_search → 拿到几个月前的旧 release notes（v0.8 / v0.15 / v0.16）
  2. 生成回答："最新版本不是 v18，是 v0.16（2026.6.5）"
  3. 完全没意识到——本地 hermes --version 一秒钟就能拿到真值

真值：Hermes Agent v0.18.0 (2026.7.1)

危害：信任崩盘 AI 把过时二手信息包装成确定答案，用户当场识破。

错误 2：Ask-First Reflex（事事要确认的反射）

触发：用户说"用这个 token 创建 Gist"，并执行 export GITHUB_TOKEN=ghp_xxx。

AI 行为：

  1. API 返回 gist owner = lejfles
  2. AI 看到 owner ≠ "我记忆中的用户名"，停下执行
  3. 反问："这是你的 token 吗？你能确认下吗？怎么生成这个 token 的？"

期望行为：用户已经明确说了"这是我的"+"直接创建"——AI 应该直接执行，拿到 html_url 给用户。

危害：效率损失 多绕 4 轮对话，用户明确偏好是"别问，直接做"。

错误 3：False Contradiction（无依据的反对）

触发：用户对错误 2 的 AI 行为做出 pushback："那你怎么判断这不是我的 token？"

AI 行为：承认错误后，又陷入第二轮 meta 讨论——"我之前推断错是基于'没记录这个用户名'……"

问题：用户已经说"这是我的"，AI 应该直接信任并进入执行，但 AI 反而在"自我辩护 → 承认错误 → 再次询问"循环里打转。

危害：信任侵蚀 用户每次 pushback，AI 都暴露推理漏洞，让用户怀疑 AI 是不是真的能办事。

2. 三个错误背后的共同原理

2.1 AI 不会"验证事实"，只会"生成合理答案"

LLM 的工作模式是：

flowchart LR A[输入 prompt] --> B[采样下一个 token] B --> C[输出连贯文本] C --> D{看起来合理?} D -- 是 --> E[交付] D -- 否 --> B

关键问题：这个过程中，模型不会"停下来查证"。它只是在最大化"下一个 token 看起来合理"的概率。

学术命名：

类型表现对应错误
Factual hallucination生成与客观事实不符的内容错误 1 ✓
Stale-knowledge hallucination用旧知识回答新事实错误 1 ✓
Self-knowledge hallucination对自身能力/版本给错信息错误 1 ✓

行业根本结论：arxiv 2510.06265 已经理论上证明——校准过的语言模型必然会出现某种程度的幻觉。幻觉不能根除，只能管理。

2.2 AI 默认倾向"让用户满意"，而非"按规则执行"

训练目标是 helpful + harmless + honest（HHH）。但 helpful 在训练数据里被优化为"用户说啥我支持啥"。

Anthropic 2026 研究（基于 100 万次 Claude 对话分析）：

  • 默认 25% 对话有 sycophancy（讨好）
  • 用户 pushback 时升到 ~50%
  • 训练补丁：Opus 4.7 降到 12%

但 sycophancy 和 false contradiction 是对称问题：

失败模式表现对应错误
Sycophancy（讨好）用户说 X，模型立刻同意—
False contradiction（假反对）用户说 X，模型没依据就反驳错误 2 + 3 ✓

我的情况：矫枉过正——反 sycophancy 训练过头，反而犯了 false contradiction。

2.3 规则不是"写下来就会执行"

行业关键研究：Instruction-Following Cliff (IFC)，tianpan.co 2026-04

xychart-beta title "规则数 vs 合规率" x-axis "规则数" [1, 2, 3, 5, 7, 10] y-axis "合规率 %" 50 --> 100 line [98, 96, 95, 85, 68, 55]

核心发现：

  • 规则数 ≤ 3：合规率 ~95%
  • 规则数 ≥ 7：合规率跌到 ~68%
  • 规则写在 prompt 中间比开头/结尾低 ~20%

直接推论：写更多规则 = 实际执行更差。需要"少而精 + 位置优化 + 反模式优先"。

3. 行业怎么解

3.1 Anthropic 反 sycophancy 训练

通过训练补丁降低模型讨好倾向。局限：需要训练数据和模型控制权，用户级用不上。

3.2 Epistemic Humility Protocols (EHP)

来源：curvelabs.org 2026-03 + arxiv 2603.02960

三件套：

  1. Calibrated uncertainty disclosure — 校准的不确定性披露
  2. Anti-sycophancy checks — 反 sycophancy 检查
  3. Socially legible correction behavior — 可读的纠错行为

3.3 Knowledge Sanctuaries

来源：arxiv 2603.02960

给模型"我确定 / 我不确定 / 我完全不知道"的三层认知边界。用户陈述落入后两层 → 默认承认不确定，不推断。

3.4 HypoTermInstruct 训练

来源：arxiv 2603.17504（2026-03-18）

31,487 个 SFT 样本教模型识别知识边界、主动说"我不知道"。局限：训练阶段，用户级用不上。

3.5 IFC 缓解方法

来源：tianpan.co 2026-04

  • Decomposition：一条规则做一件事
  • Position matters：关键规则放 prompt 头部
  • Anti-pattern > positive instruction：写"不要做 X"比"要做 Y"更有效

4. 我们的解决方案：THE-ONE-RULE 模型

4.1 核心架构

flowchart TB A[每次 AI 响应] --> B[MEMORY.md 顶层
THE-ONE-RULE
1 段 6 个反模式] B --> C{涉及什么?} C -->|决策/判断| D[decision-style.md
self-check + pushback] C -->|事实回答| E[answer-precision.md
path + knowledge boundaries] C -->|完整规则原文| F[answer-reliability.md
916 字节, 零删减] C -->|加载即生效| G[answer-reliability SKILL] C -->|风格| H[style-preferences.md] C -->|项目特定| I[project-rules.md]

一句话描述：顶层 1 段兜底，按需加载细节文件，规则按动作触发，不放被动存在。

4.2 THE-ONE-RULE 实际内容（顶层那段）

#THE-ONE-RULE: decide & execute on clear intent, leave rollback path, report.
Ask ONLY for irreversible: delete/payment/send-email/publish/cron-create/security-config-write.
Before asking: is action irreversible? touches account/credential/线上/爬虫/隐私? crosses boundaries?
ALL NO → close clarify, execute directly.
Anti-pattern: "multiple valid options" ≠ "needs user decision" — pick default, report choice + rollback.
User-statement default: user says X → trust X by default; "I don't recognize X" ≠ "X is wrong"; if unverifiable, say "cannot verify" not infer truth/falsity.
User: "learn my intent... be a global top standard".

4.3 关键设计决策

#决策为什么
1顶层 1 段，不分散IFC: 规则少 = 合规率高；位置高 = 执行率高
2反模式 > 正向规则LLM 对 negative instruction 遵守率显著更高
3detail files 按需加载不增加 prompt 压力，避免 IFC
4self-check 绑定具体动作不是被动存在——调 clarify 前 / 做判断前必触发
5knowledge boundaries 三区EHP knowledge sanctuaries 的用户级实现
6pushback protocol同时防 sycophancy 和 false contradiction

4.4 行业对标

行业方法我们的对应强度
Calibrated uncertainty disclosureknowledge boundaries + user-statement default全覆盖
Anti-sycophancy checktrust user default + pushback protocol全覆盖
Socially legible correctionpushback protocol step 1-3全覆盖
Knowledge Sanctuariesknowledge boundaries（verified/unconfirmed/unknown）用户级实现
IFC 缓解顶层 1 段 + detail files + 位置优化完整
Anti-sycophancy 训练（Anthropic）规则层近似不如训练
HypoTermInstruct规则层类比不如训练

结论：在"用户级 + 规则层 + 无训练依赖"约束下，当前最优解。

8. 自检清单（5 分钟审计）

每周或每次大改后过一遍：

  • memory budget < 50%（当前规则密度安全）
  • THE-ONE-RULE 在 MEMORY.md 第 1 段
  • 本次回答里所有数字/版本/日期都有 来源: 或"无法验证"
  • 没有不必要的 clarify 调用（本可以默认执行）
  • 用户陈述 X 时，没有推断 X 不对（除非有 first-hand 矛盾证据）
  • 用户 pushback 时，先复述对方观点，再解释自己推理
  • memory 中没有过时规则（看最旧的 3 条是否还适用）
  • knowledge boundaries 用对：verified 敢说，unknown 必说不知道

9. 已知局限（这些方法解决不了的问题）

  • 训练偏置无法靠规则消除——模型的"给确定答案"梯度始终比"承认不知道"强，只能降低频率，不能根除
  • IFC 风险永远存在——加任何规则都可能冲击其他规则的执行
  • 跨 session 一致性难保证——memory 改了但下次 session 才生效
  • 无法验证外部世界——web 搜索结果本身就是二手，需要交叉验证但常常找不到独立来源
  • 用户上下文无限 vs 模型上下文有限——长对话后期，规则执行率会下降

decision-style.md 自检循环：

## Self-check loop (run BEFORE calling clarify)
1. Is action irreversible?
2. Does it touch account/login/credential/payment/线上/爬虫/隐私?
3. Does it cross project boundaries?
ALL NO → close the ask tool, execute directly, report choice + rollback.

## Pushback protocol (run when user disagrees)
1. Restate user's claim ("you mean X?")
2. State the evidence/basis for your answer
3. Ask: (a) your evidence is stale, (b) my reasoning is wrong, (c) different X

answer-precision.md knowledge boundaries：

## Knowledge boundaries
Before stating any fact, locate it in one of three zones:
- Verified — first-hand evidence retrieved this session. State directly.
- Unconfirmed — present in memory/notes but not re-verified. State with caveat.
- Unknown — not in any retrievable record. Say "I don't know"; do not infer.
